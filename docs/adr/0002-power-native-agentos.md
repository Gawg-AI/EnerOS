# ADR-0002：定位为电力原生 Agent 操作系统

- **状态**：Accepted
- **日期**：2026-06-21
- **决策者**：EnerOS 核心团队
- **相关 ADR**：[ADR-0001](./0001-record-architecture-decisions.md)

## 上下文

电力与能源领域的 AI Agent 应用面临根本性困境：通用 Agent 框架（如 LangChain、AutoGen 等通用编排框架）缺乏对电力系统物理规律的原生理解，安全约束被降级为提示词级别的建议，电网拓扑与电气耦合关系被忽视，设备异构性带来的协议差异难以统一调度。

在通用 IT OS 或通用 Agent 框架上"外挂"电力知识的方案存在以下问题：

1. **物理盲区**：Agent 不理解潮流、电压、频率等物理量，无法判断决策的物理可行性
2. **约束缺失**：安全约束（N-1、热稳定、电压越限）被当作"提示词"而非系统级硬约束，可被绕过
3. **拓扑无感**：电网被视为扁平数据，无法感知拓扑结构与电气耦合关系
4. **时序割裂**：电力系统是强时序耦合系统，通用框架缺乏时间维度的一等公民支持
5. **设备异构**：变压器、断路器、逆变器各有独立模型与协议（IEC 104/61850/Modbus/DNP3/GOOSE/SV），难以统一调度
6. **实时性不足**：继电保护动作需毫秒级完成，通用 OS 内核无法提供硬实时保证

我们需要决定 EnerOS 的核心定位：是构建一个通用 OS 上的电力应用框架，还是从电力原生出发构建操作系统。

## 决策

**将 EnerOS 定位为电力原生 Agent 操作系统（Power-Native AgentOS），以电力系统领域模型为核心，Agent 作为 OS 调度单元。**

具体含义：

1. **电力领域模型作为 OS 原生抽象**：电力拓扑、潮流计算、设备模型不是外挂插件，而是操作系统内核的原生抽象（L0 基础层）。Agent 从诞生起就运行在电网的物理世界模型之上。

2. **Agent 作为 OS 调度单元**：每个 Agent 是一个独立进程，由 OS 内核（eneros-os/agentos）调度，而非应用层的协程或线程。Agent 对应电网中的功能节点（厂站、馈线、设备），天然具备拓扑感知能力。

3. **安全约束作为内核法律**：安全约束（N-1 校验、热稳定、电压限值）由内核（eneros-constraint + eneros-gateway）强制执行，任何 Agent 的决策不可逾越物理可行域。安全不是提示词，而是操作系统级的硬约束。

4. **电力协议走 OS 原生能力**：GOOSE/SV 等二层协议通过 AF_PACKET 原始套接字实现，作为 OS 原生能力而非用户态库。IEC 104/61850/Modbus 等协议适配器位于 L2 能力层，由设备管理器统一调度。

5. **双执行架构**：通用执行域承载 Agent 编排与 AI 推理，实时执行域保障保护逻辑与开关操作的确定性时延。安全域不可被通用域阻塞。

6. **时间维度作为一等公民**：时序数据（eneros-timeseries）作为基础层能力，支持实时数据流、历史回溯与预测推演的原生操作。

## 后果

### 正面

- Agent 天然"懂电"，无需通过提示词注入电力知识，决策的物理可行性由内核保障
- 安全约束不可绕过，满足电力系统对安全性的刚性需求
- 电力协议作为 OS 原生能力，性能与可靠性优于用户态实现
- 双执行架构兼顾 AI 推理的灵活性与保护逻辑的确定性
- 领域模型内建使系统对电力场景的适配更自然，减少"外挂"带来的阻抗失配

### 负面

- 系统复杂度高于通用框架，开发与维护成本更高
- 电力领域知识与 OS 内核耦合，非电力场景的复用性降低
- 需要开发者同时理解电力系统与操作系统两个领域
- 双执行架构增加了跨域通信的复杂度

### 中性

- 定位明确为电力能源场景，不追求通用性，这是有意识的范围收窄
