# Phase 1 蓝图规划（v0.23.0 ~ v0.74.0）Spec

## Why

Phase 0 的 21 个版本蓝图（v0.2.0 ~ v0.22.0）已全部生成完毕，每个版本 20-30 页极详尽蓝图。Phase 1 "单机 MVP" 是 EnerOS 从内核基础设施走向完整端到端场景的关键阶段，共 52 个版本（v0.23.0 ~ v0.74.0），需要同样极详尽的版本蓝图来指导工程实现。Phase 1 的出口标准是"一台 Edge Box 端到端跑通市场信号 → LLM → Solver → 控制 → 储能链路，双脑链路 < 2s，比传统 EMS 收益 ≥ 10%"。

## What Changes

- 为 Phase 1 的 52 个版本（v0.23.0 ~ v0.74.0）生成极详尽蓝图 MD 文件，每个文件 20-30 页
- 蓝图覆盖 12 个子模块：P1-A 存储与文件系统、P1-B 网络协议栈、P1-C 密码学安全、P1-D Agent Runtime、P1-E System Agent、P1-F 设备协议栈、P1-G 四遥与SOE、P1-H RTOS组件、P1-I LLM推理、P1-J Solver、P1-K 双脑协同、P1-L MVP集成
- 每个蓝图包含：版本目标、前置依赖、交付物清单、详细设计（数据结构/接口/算法/错误处理）、实现计划、测试计划、验收标准、风险与注意事项
- 文件输出到 `e:\eneros\蓝图\versions\` 目录

## Impact

- Affected specs: 无前序 spec（首次创建）
- Affected code: 不涉及代码修改，仅生成蓝图文档
- 前序依赖: Phase 0 蓝图（v0.2.0~v0.22.0 已完成）
- 后续影响: 生成的蓝图将指导 Phase 1 全部 52 个版本的工程实现

## ADDED Requirements

### Requirement: Phase 1 版本蓝图文件生成

系统 SHALL 为 Phase 1 的每个版本（v0.23.0 ~ v0.74.0，共 52 个版本）生成独立的极详尽蓝图 MD 文件。

#### Scenario: 蓝图文件完整性
- **WHEN** Phase 1 蓝图生成完成
- **THEN** `e:\eneros\蓝图\versions\` 目录下存在 52 个 MD 文件，文件名格式为 `v0.XX.0_版本名称.md`

#### Scenario: 蓝图内容结构
- **WHEN** 打开任意 Phase 1 版本蓝图文件
- **THEN** 文件包含完整的 0~9 章结构（文档元信息/版本目标/前置依赖/交付物清单/详细设计/实现计划/测试计划/验收标准/风险注意事项/附录）

#### Scenario: 蓝图详细程度
- **WHEN** 审查任意 Phase 1 版本蓝图文件
- **THEN** 文件行数在 2500-4000 行之间（约 20-30 页），包含完整的 Rust trait/struct 定义、接口签名、核心算法伪代码、测试用例

#### Scenario: 子模块完整性
- **WHEN** 统计生成的蓝图文件
- **THEN** 12 个子模块的每个版本都有对应的蓝图文件，无遗漏

### Requirement: 关键依赖链覆盖

系统 SHALL 确保 Phase 1 的 6 个关键瓶颈版本蓝图特别详尽：

1. **v0.24.0 文件系统** — 无 FS 则模型加载/配置/日志无法做
2. **v0.28.0 TCP/IP** — 无网络栈则所有网络协议无法做
3. **v0.31.0 国密** — 无密码学则能力模型/mTLS 无法做
4. **v0.59.0 LLM引擎** — 双脑链路起点
5. **v0.71.0 双脑联调** — 双脑链路 < 2s 验证
6. **v0.74.0 MVP集成** — Phase 1 出口标准验证

#### Scenario: 瓶颈版本蓝图质量
- **WHEN** 审查 6 个关键瓶颈版本蓝图
- **THEN** 每个文件不少于 3000 行，包含完整的端到端验证方案和性能目标

### Requirement: Phase 1 出口标准验证

v0.74.0 蓝图 SHALL 包含 Phase 1 出口标准的逐项验证方案：

1. 真实储能场景 autonomous 运行
2. 双脑链路延迟 < 2s
3. 比传统 EMS 收益 ≥ 10%

#### Scenario: 出口标准覆盖
- **WHEN** 审查 v0.74.0 蓝图
- **THEN** 文件包含 Phase 1 出口标准验证报告，逐项列出验证方法和通过条件
