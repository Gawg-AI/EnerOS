## 一、每次开发前的检查清单

### 1.1 开机必做（5 项）

- [ ] 确认当前所在分支：`git branch`（禁止在 main 上直接开发）
- [ ] 确认工作区干净：`git status`（无未提交的临时改动）
- [ ] 拉取最新代码：`git pull --rebase origin <当前分支>`
- [ ] 确认 Rust 工具链版本：`rustup show`（必须与 `rust-toolchain.toml` 一致）
- [ ] 确认当前开发版本号（对照路线图 `蓝图/Power_Native_Agent_OS_Version_Roadmap_v3.md`）

### 1.2 开发中必守（8 项）

- [ ] **所有 Rust 代码必须 no_std**（蓝图 §43.1 硬性要求，覆盖全项目）
- [ ] 禁止 `use std::*`，改用 `alloc::*` / `core::*` / `heapless::*` / `spin::*`
- [ ] **新 crate 必须放入 `crates/<subsystem>/`**（见 §2.3.1），禁止直接放根目录
- [ ] **新文档必须放入 `docs/<topic>/`**（见 §2.3.3），禁止平面化放 `docs/` 根
- [ ] 提交前运行：`cargo fmt` + `cargo clippy` + `cargo test`
- [ ] 提交信息遵循 Conventional Commits（见下文 §六）
- [ ] **绝不提交**：`target/`、`*.elf`、`*.bin`、`*.dtb`、IDE 缓存、密钥
- [ ] 每个版本开发完成后更新该版本的 checklist

### 1.3 收工必做（5 项）

- [ ] **目录结构校验**：执行 §2.4 校验清单（C1~C15 逐项检查，新增 crate/文档时必查）
- [ ] 清理构建产物：`cargo clean`（除非需要保留增量编译缓存）
- [ ] 检查 `.gitignore` 是否覆盖了新产生的文件类型
- [ ] 运行 `git status` 确认无垃圾文件被追踪
- [ ] 推送前确认 CI 能通过：本地先跑 `make ci-local`（若存在）

## 二、仓库目录结构规范（★ 强制执行）

> ⚠️ **本章节为硬性规则，所有开发必须严格遵守。代码完成后必须按 §2.4 校验清单逐项检查。**
>
> **设计原则**：本项目共 205 个开发版本，每个版本可能新增 crate。为避免根目录 crate 膨胀到上百个文件夹，所有 Rust crate 必须按子系统分组归入 `crates/<subsystem>/` 下；所有工程文档必须按方向分类归入 `docs/<topic>/` 下。

### 2.3 新建文件夹原则（强制）

#### 2.3.1 Crate 分组规则

1. **所有 Rust crate 必须放入 `crates/<subsystem>/` 下**，禁止直接放在仓库根目录
   - 子系统划分：`kernel` / `hal` / `runtime` / `drivers` / `ai` / `protocols` / `agents`
   - 子系统目录名：全小写 + 下划线（如 `device_drivers`），不用驼峰
   - crate 目录名：与 Cargo 包名去掉 `eneros-` 前缀后的部分一致（如 `eneros-time` → `crates/drivers/time/`）
2. **例外**：`ci/` 作为 CI 工具 crate，独立于 `crates/` 放在根目录（因属构建工具，非产品代码）
3. **嵌套深度**：`crates/<subsystem>/<crate>/src/<module>.rs` 为最大 4 层；`crates/kernel/heap/src/impl/buddy/core.rs` 过深
4. **跨 crate 引用路径**：使用相对路径 `path = "../../<subsystem>/<crate>"`，禁止绝对路径
5. **禁止中文目录名**：所有代码相关目录用英文
6. **蓝图目录例外**：`蓝图/` 是唯一允许的中文目录名（文档专用）

#### 2.3.2 子系统归属判定（新增 crate 时必读）

新增 crate 时，按以下规则判定归属子系统：

| 子系统 | 归属判定 | 示例 |
|--------|---------|------|
| `crates/kernel/` | 内核态运行、调度、内存管理、Panic、SMP | kernel, mm, heap, sched, smp, panic |
| `crates/hal/` | 硬件抽象层、板级支持、CPU 核心驱动 | hal, board |
| `crates/runtime/` | 用户态运行时、seL4 绑定、用户态组件 | runtime, sel4-sys, hello, user/* |
| `crates/drivers/` | 外设驱动（时钟、看门狗、UART、GPIO 等） | time, watchdog |
| `crates/ai/` | AI Runtime（LLM 推理、Solver） | （Phase 2+） |
| `crates/protocols/` | 设备协议栈（Modbus/IEC104/61850/CAN） | （Phase 2+） |
| `crates/agents/` | Agent 实现 | （Phase 2+） |

> **不确定时**：查阅蓝图对应版本的 §交付物章节，按蓝图声明的子系统归属。

#### 2.3.3 文档分类规则（强制）

1. **所有工程文档必须按方向分类归入 `docs/<topic>/` 下**，禁止平面化堆放在 `docs/` 根目录
2. 文档方向划分：

| 文档方向 | 归属 | 示例文档 |
|---------|------|---------|
| `docs/hal/` | HAL 设计、硬件抽象、CPU/中断/定时器/页表 | hal-design-whitepaper.md, gicv3-driver-guide.md |
| `docs/kernel/` | 内核堆、内存管理、分区隔离 | kernel-heap-design.md, partition-isolation-design.md |
| `docs/runtime/` | 用户态运行时、seL4 绑定、用户堆 | userland-runtime-design.md, user-heap-design.md |
| `docs/drivers/` | RTC、看门狗、定时器、喂狗协议 | rtc-driver-design.md, watchdog-design.md |
| `docs/smp/` | 多核启动、IPI、调度器、核绑定 | smp-boot-design.md, multi-core-scheduler-design.md |
| `docs/boot/` | 硬件启动、设备树、串口调试 | hardware-boot-guide.md, device-tree-spec.md |
| `docs/conventions/` | 代码规范、提交规范 | code-conventions.md, commit-conventions.md |
| `docs/ci/` | CI/CD 流水线文档 | ci-cd-manual.md |

3. **新增文档时**：优先放入已存在的对应方向子目录；若无匹配方向，新建子目录（全小写+下划线）
4. **禁止**：在 `docs/` 根目录直接放 `.md` 文件（除 `README.md` 索引文件外）

### 2.4 代码完成后校验清单（★ 强制执行）

> **每次开发完成、提交前必须逐项检查。未通过校验禁止提交。**

#### 2.4.1 目录结构校验

- [ ] **C1 新 crate 位置**：新 crate 是否放入 `crates/<subsystem>/` 下？未直接放根目录？
- [ ] **C2 workspace members**：根 `Cargo.toml` 的 `members` 是否已添加新 crate 路径（如 `"crates/drivers/xxx"`）？
- [ ] **C3 跨 crate path 引用**：所有 `Cargo.toml` 中的 `path = "..."` 是否使用正确的相对路径？
- [ ] **C4 文档分类**：新增的 `.md` 文档是否放入 `docs/<topic>/` 子目录？未平面化放 `docs/` 根？
- [ ] **C5 无根目录 crate**：仓库根目录下无除 `ci/` 外的 Rust crate 文件夹？

#### 2.4.2 构建校验（必须全部通过）

```bash
# 1. workspace 能解析所有成员
cargo metadata --format-version 1 > /dev/null

# 2. 主机侧测试
cargo test --workspace --exclude eneros-kernel --exclude eneros-hello

# 3. HAL mock 测试
cargo test -p eneros-hal --features mock

# 4. 交叉编译验证（所有 crate）
cargo build -p eneros-<新crate> --target aarch64-unknown-none -Z build-std=core,alloc -Z build-std-features=compiler-builtins-mem

# 5. 格式与 lint
cargo fmt --all -- --check
cargo clippy --workspace --exclude eneros-kernel --exclude eneros-hello --all-targets -- -D warnings

# 6. 安全扫描
cargo deny check advisories licenses bans sources
```

| 目标规格 | `.cargo/config.toml` `*.json`（target spec） | 交叉编译配置 |

### 3.3 大文件处理

- **模型文件**（`.gguf` `.bin` 权重）：**绝不入仓**，用 Git LFS 或外部存储
- **seL4 源码**：作为 git submodule 或独立下载，不入主仓
- **测试固件**：超过 1MB 的二进制测试数据用 Git LFS

### 3.4 分支保护与清理

- `main` 分支：受保护，只接受 PR 合并，必须 CI 全绿
- `develop` 分支：集成分支
- `feature/v0.XX.0-xxx`：功能分支，合并后删除
- 定期清理已合并的远程分支：`git remote prune origin`

### 4.2 GPU 优先测试规则（蓝图 §43.3）

> ⚠️ 本规则**仅适用于**：模型训练（云端）、模型量化校准、数字孪生仿真加速。
> **不适用于**：边缘推理（用 llama.cpp C 推理）、RTOS 控制路径、Solver 求解。
> **设计评审 P1-2 修正**：边缘 LLM 推理统一采用 llama.cpp（C API），禁止在边缘侧使用 PyTorch。RL/MARL/在线学习已后置为研究附录（ADR-0002）。

适用场景规则：
- 模型训练/校准时：`model.to("cuda")`，`with torch.no_grad():`，GPU 不可用退 CPU
- 边缘 LLM 推理：有 NPU/GPU → llama.cpp GPU 后端；无 → CPU + INT4 量化
- Solver（LP/MILP）：CPU 求解，不涉及 GPU

### 4.3 no_std 合规性检查（蓝图 §43.1）

> **设计评审 P1-1 要求**：Phase 1~4 蓝图中所有 `std::` 违规已清理完毕。开发中新增代码必须严格遵守 no_std，CI 通过 clippy 强制拦截。

**适用范围：全项目所有 Rust 代码必须 no_std**，分三层：

| 层级 | 范围 | no_std | alloc | 说明 |
|------|------|--------|-------|------|
| ① 内核态/RTOS 态 | Phase 0~3 | ✅ 必须 | 视场景 | 无堆用 `heapless`，有堆用 `alloc` |
| ② Agent Runtime | Phase 1 起（管理信息大区） | ✅ 必须 | ✅ 可用 | 依赖 v0.11.0 用户堆；网络用 smoltcp，不用 `std::net` |
| ③ LLM Rust 封装层 | v0.59.0 | ✅ 必须 | ✅ 可用 | C 底层（llama.cpp）例外，可有自己的运行时 |

```rust
// ❌ 禁止（全项目所有 Rust 代码）
use std::collections::HashMap;
use std::time::Duration;
use std::sync::Mutex;
use std::net::TcpStream;

// ✅ 正确
use alloc::collections::BTreeMap;   // 需 alloc crate + v0.11.0 用户堆
use core::time::Duration;
use spin::Mutex;                    // no_std 自旋锁
use heapless::FnvIndexMap;          // 无堆场景
// 网络用 smoltcp，不用 std::net
```

### 4.4 代码示例可用性要求（蓝图 §43.2）

- **瓶颈版本**（★ 标记，如 v0.10.0/v0.12.0/v0.16.0/v0.22.0）：代码必须"骨架可用"，不能 stub
- **非瓶颈版本**：可用伪代码，但 trait/struct 签名必须可编译
- **架构相关代码**：必须与声称的架构一致（GICv3 不能用 GICv2 代码冒充）
- **9 节版本模板**：每个版本蓝图必须包含 9 节（版本目标/前置依赖/交付物清单/详细设计/技术交底/测试计划/验收标准/风险/多角度要求），缺节视为不合格

### 5.5 默认集成清单（防"自研一切"，蓝图 §42.4）

> **设计评审 P1-3 要求**：优先集成成熟开源组件，禁止重复造轮子。自研仅限"能源行业特有 + 无开源替代"的场景。

| 子系统 | 默认集成 | 自研范围 | 理由 |
|--------|---------|---------|------|
| 文件系统 | **littlefs2**（掉电安全、磨损均衡） | 仅封装能源特有点表存储接口 | 成熟开源，工业级验证 |
| TCP/IP 栈 | **smoltcp**（no_std） | 无 | 唯一成熟 no_std 网络栈 |
| DDS 总线 | **Cyclone DDS** | 仅 Rust 封装层 | OMG 标准，工业级实现 |
| MILP/LP Solver | **HiGHS** | 仅 Rust FFI 封装 + 能源建模层 | 开源高性能，SCIP 许可证风险已排除 |
| LLM 推理 | **llama.cpp**（C API） | 仅 Rust 封装层 | 边缘推理首选，INT4 量化支持 |
| 密码学 | **国密 SM2/SM3/SM4** 硬件加速 | 协议层适配 | 硬件加速 + 合规要求 |

### 5.6 内存预算（蓝图 §43.6，每分区量化）

> **设计评审 P1-4 要求**：每个分区必须有量化内存预算，OOM 阈值明确。开发中新增组件必须声明内存占用。

| 分区 | 预算 | OOM 策略 | 说明 |
|------|------|---------|------|
| RTOS 控制大区 | ≤ 32 MB | 硬实时不分配堆，纯静态 | 10ms 控制周期，零 GC |
| Agent Runtime（管理信息大区） | ≤ 64 MB | 降级到规则引擎 | v0.11.0 用户堆配额管理 |
| LLM 7B INT4 | ≤ 4 GB | 降级到 Solver-only（L1 路径） | llama.cpp 内存映射加载 |
| Solver（LP/MILP） | ≤ 128 MB | 缩减问题规模/超时降级 | HiGHS 求解器 |
| 文件系统缓存 | ≤ 16 MB | LRU 淘汰 | littlefs2 |
| OOM 阈值 | 总用量 > 90% | 触发 OOM handler：冻结非关键 Agent | 蓝图 §43.6 |

### 5.7 合规矩阵与 SBOM（蓝图 §43.7~43.8）

> **设计评审 P1-6/P1-7 要求**：合规与许可证/供应链风险必须持续追踪。

- **合规矩阵**（§43.7）：横向隔离（36 号文）/ 纵向加密认证 / 等保 2.0 三级 / 信创认证 / 国密合规 / 电力行业准入 — 每项对应具体版本与验收条件
- **SBOM 清单**（§43.8）：所有第三方依赖（含 C 库）必须登记许可证、来源、已知 CVE；CI 中 `cargo deny check` 强制扫描
- **LLM 必要性证据**（P1-5）：每个引入 LLM 的场景必须有"为何 Solver 不够"的论证，记录于版本蓝图 §技术交底

## 八、常见陷阱与禁忌

### 8.1 绝对禁忌

| # | 禁忌 | 后果 |
|---|------|------|
| 1 | 提交 `target/` 目录 | 仓库膨胀数 GB，clone 极慢 |
| 2 | 提交 `.env` / 密钥文件 | 安全漏洞 |
| 3 | 内核代码用 `std::` | 违反 no_std，编译失败 |
| 4 | 在 main 分支直接提交 | 破坏版本管理 |
| 5 | 跳过 CI 直接合并 | 质量失控 |
| 6 | 提交二进制镜像 `.elf/.bin` | 仓库膨胀 |
| 7 | GICv3 代码用 GICv2 实现 | 架构不一致，运行异常 |
| 8 | 提交 IDE 缓存（.idea/.vscode） | 冲突频发 |
| 9 | **crate 直接放根目录**（非 `crates/<subsystem>/`） | 根目录膨胀到上百文件夹，无法维护（见 §2.3.1） |
| 10 | **文档平面化放 `docs/` 根** | 34+ 文档无法分类检索（见 §2.3.3） |
| 11 | **跳过 §2.4 校验清单提交** | 目录结构失控，跨 crate 引用断裂 |
| 12 | **违反 ADR 决策**（如 Phase 3 重新引入自研内核、v1.0.0 加入研究特性） | 架构方向偏移，版本账本混乱（见 §5.4） |
| 13 | **边缘 LLM 用 PyTorch** | 违反 no_std + 内存爆炸；边缘推理必须用 llama.cpp（见 §4.2） |
| 14 | **重复造轮子**（自研已有成熟开源替代的组件） | 浪费工期，质量不可控（见 §5.5 默认集成清单） |
| 15 | **缺少内存预算声明** | OOM 不可控，分区隔离失效（见 §5.6） |
| 16 | **版本蓝图缺节**（少于 9 节） | 交付物定义不清，验收无依据（见 §4.4） |

### 8.2 常见问题处理

**Q: `target/` 已被提交怎么办？**
```
git rm -r --cached target/
echo "target/" >> .gitignore
git commit -m "chore: 移除误提交的 target/ 并加入 gitignore"
```

**Q: Cargo.lock 该入仓吗？**
- 二进制 crate（本项目）：✅ 入仓
- 库 crate：❌ 不入仓

**Q: seL4 源码怎么管理？**
- 作为 git submodule，或
- 构建脚本自动下载（`tools/setup-toolchain.sh`），不入主仓
